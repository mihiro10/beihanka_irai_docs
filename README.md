## 📌 プロジェクトの背景と目的 (Why does this system exist?)
本システムは、食品製造工場において**「惣菜課（生産管理部門）」と「米飯課（製造現場）」の間で発生していたアナログな課題を解決し、製造依頼を一元管理・可視化するため**に開発されたAppSheetアプリケーションです。

ミスや無駄なやり取りを極限まで減らすため、以下の明確な目標を持って導入されました：

### 🛑 解決した現場の課題 (Pain Points Resolved)
1. **情報の一元管理と抜け漏れ防止**
   * *課題:* 人の記憶やFAX送信に依存しており、「出した／受け取った」の状況が不透明だった。
   * *解決:* いつ誰が見ても同じ状況が把握できる完全な一元管理ダッシュボードを構築。
2. **転記ミスの撲滅**
   * *課題:* 紙の製造依頼書を裏紙に手書きで書き写す作業が発生し、ミスを誘発していた。
   * *解決:* iPadのデジタルデータを直接見て製造・承認するフローに変更し、書き写しをゼロに。
3. **適正在庫と優先順位の明確化**
   * *課題:* 「優先度が高いもの」ではなく「来た順」や現場都合で作ってしまうケースがあった。
   * *解決:* システム上で優先度の高いものから着手できるようにし、残業の必要性を論理的かつ明確に判断可能に。
4. **急な「後出し変更」と無駄なやり取りの削減**
   * *課題:* 現場の生産能力が見えないまま依頼され、電話・口頭での重複確認や無理な依頼が多発していた。
   * *解決:* 依頼時点で現場の空き時間を可視化。無理な依頼や直前の変更をシステム上で防ぐルールを実装。

## ⚙️ コアとなるビジネスロジック (System Rules)
上記の課題を解決するため、システムには以下の強力なロジックが組み込まれています：
*   **キャパシティ管理:** 1日あたり「早朝」「昼間」の2つの時間帯で上限分数（例: 180分）を設定し、空きキャパを可視化。
*   **米飯課優先の予定ブロック:** 米飯課が事前に「清掃」などを登録すると、生産管理側の依頼可能枠が動的に減少。
*   **4日ルールのロック機能:** 製造日の4日以内に入った依頼は米飯課の承認により**【確定】**状態となり、編集・削除が自動ロックされる（直前の後出し変更を強制ブロック）。

## 🎯 このGitHubリポジトリの役割 (Why does this repo exist?)
**AppSheet自体には「ソースコード」や「Gitを用いたバージョン管理」の概念が存在しません。**
そのため、数年後にこのアプリの改修を行おうとしたエンジニア（またはAIアシスタント）が、「どの数式がどこに影響しているのか」「このSlice（仮想ビュー）やActionは何のために動いているのか」という全体設計の全貌を見失う致命的なブラックボックス化問題が発生します。

本リポジトリは、**「AppSheetアプリの設計書・仕様・ビジネスロジックをGitで完全にバージョン管理し、常に最新のシステムアーキテクチャを未来の担当者へ安全に引き継ぐこと」**を唯一の目的として作られた特殊形態のリポジトリです。

## 📂 ディレクトリ構成 (Repository Structure)

*   **[`/user_guides/`](user_guides/)**
    *   現場のタブレット操作者（生産管理・米飯課）に向けた純粋な運用マニュアルです。技術的な文脈を排除した操作手順書が保管されています。
*   **[`/appsheet_architecture/`](appsheet_architecture/)**
    *   このアプリの論理的な設計書群です。データベースのテーブル定義、仮想カラム式、UIレイアウト（Views）、および自動化ルール（Actions/Slices）が、すべての依存関係と共に記録されています。改修を行うエンジニアはまずここを読んでください。
*   **[`/automation_tools/`](automation_tools/)**
    *   AppSheet側のGUI変更結果を、GitHubの仕様書へ自動同期するための専用のPythonツール群です（後述）。
*   **[`/archive/`](archive/)**
    *   過去の要件定義書や、検証用に抽出した過去データ（JSON形式の各テーブルバックアップ）などのアーカイブ置き場です。

## 🛠 開発者向け: リポジトリの運用方法 (How to work with this repo)
新しい開発者がこのアプリを適切に改修・メンテナンスする際のワークフローは以下の通りです。

1.  **① 既存仕様・影響範囲の把握**  
    AppSheetのエディタを開く前に、まずは `appsheet_architecture/` 内の設計書を読み込み、目的の変更箇所が他にどう影響するかを完全に把握してください。
2.  **② AppSheetエディタでのGUI改修**  
    実際にAppSheetのブラウザ上で設定や数式を変更します。
3.  **③ 仕様書（当リポジトリ）への自動同期**  
    変更後、必ず **[`automation_tools/README.md`](automation_tools/README.md)** の手順に従い、専用のPythonスクリプトを実行してリポジトリの設計書を最新化し、Gitにコミットしてください（これにより、過去の手書きコメントを保持したままテーブル図等を最新に再構築できます）。
